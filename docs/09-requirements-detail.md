# 需求细化

## 1. 账号和登录需求

系统内存在三类需要登录或认证的主体：

- Web 管理端用户。
- 移动客户端用户。
- Agent 本地交互用户。

登录要求：

- Web 和移动端使用 OIDC 登录。
- Agent 首次绑定时也必须触发用户登录或设备授权流程。
- Agent 的设备身份和用户登录身份分离：
  - 设备身份用于 Agent 长连接、会话上报、审批决策投递。
  - 用户身份用于确认谁绑定了设备、谁在 Agent 本地处理了审批。
- 服务端不允许未绑定设备上报会话或审批。
- 服务端不允许未登录客户端查看设备、会话或审批。

## 2. 设备绑定需求

一个账号可以绑定多台机器，每台机器运行一个或多个 Agent 实例时，默认只允许一个主 Agent 负责审批采集。

绑定规则：

- 设备必须归属于一个租户。
- 设备必须有一个绑定发起用户。
- 设备可以授权给同租户内多个用户查看或审批。
- 设备禁用后，所有会话上报、审批投递和本地处理同步停止。
- 设备令牌只代表机器身份，不代表某个用户。

绑定方式：

- Web/移动端生成激活码，Agent 输入激活码完成绑定。
- Agent 本地发起登录，用户选择租户后完成绑定。
- MVP 至少支持激活码绑定；Agent 本地 OIDC 登录绑定作为增强能力。

## 3. 客户端实例需求

同一用户可以同时登录多个客户端：

- 多台手机。
- 多个浏览器。
- Agent 本地 UI。

服务端需要记录客户端实例，用于推送和同步：

- 客户端类型：`web`、`mobile_ios`、`mobile_android`、`agent_desktop`。
- 登录用户。
- 所属租户。
- 推送 token，移动端必需，Web/Agent 可选。
- WebSocket 在线状态。

通知规则：

- 同一个审批请求要通知所有有权限的在线客户端实例。
- 移动端离线时通过 Push 提醒。
- Web 和 Agent 本地 UI 在线时通过 WebSocket 提醒。
- 任意一个端处理审批后，所有其他端必须收到状态更新。

## 4. 多机器和多 CLI 需求

一个账户支持多台机器：

- 用户可以在移动端查看自己有权限的全部 Agent 机器。
- 用户可以进入某台机器查看该机器所有 AI CLI 会话。
- 用户可以按 CLI 类型、会话状态、时间筛选。

一台机器支持多种 AI CLI：

- Codex。
- Claude Code。
- openCode。
- GitHub Copilot CLI。
- Gemini CLI。
- 自定义 CLI。

同一台机器可以同时存在多个 CLI 会话：

- 同一种 CLI 多个并发会话。
- 不同 CLI 多个并发会话。
- 每个会话独立产生审批事件。
- 审批事件必须明确归属 `device_id + session_id + cli_type`。

## 5. 会话查看需求

移动端必须支持通过服务端查看 Agent 所在机器的会话信息。

设备详情页需要展示：

- 设备名称。
- 平台和架构。
- Agent 版本。
- 在线状态。
- 最近心跳时间。
- 当前活跃会话数。
- 今日审批数和待审批数。

会话列表需要展示：

- CLI 类型。
- 会话状态。
- 启动时间。
- 结束时间。
- 最近输出摘要。
- 待处理审批数量。
- 最近审批状态。

会话详情需要展示：

- CLI 类型。
- 会话状态。
- 脱敏后的启动命令。
- 脱敏后的最近上下文。
- 关联审批列表。
- 投递和回写状态。

安全要求：

- 默认不展示完整终端输出。
- 只展示脱敏后的输出摘要和审批上下文。
- 完整输出回放必须由租户管理员启用。

## 6. 审批通知和同步需求

审批创建后：

- 服务端创建审批单。
- 服务端通知所有有权限的客户端实例。
- 移动端收到 Push 后必须通过 API 拉取详情。
- Web、移动端、Agent 本地 UI 都可以展示审批详情。

审批处理后：

- 第一个成功提交决策的端成为处理方。
- 服务端记录处理方用户、客户端类型、处理方式、处理内容。
- 其他端收到 `approval.updated` 实时消息。
- 其他端界面必须从待处理变成已处理，并展示处理方和处理结果。

需要展示的处理信息：

- 处理用户。
- 处理客户端类型：手机、Web、Agent 本地。
- 处理时间。
- 处理动作：批准、拒绝、自定义回复、超时拒绝、策略处理、本地接管。
- 自定义回复内容，必须脱敏。
- 投递状态：待投递、已发送、已回写、失败。

并发规则：

- 多端同时处理同一审批时，只允许一个成功。
- 失败端需要显示“已由其他端处理”，并展示最新处理结果。
- 相同端重试同一请求必须幂等。

## 7. Agent 本地提醒和处理需求

Agent 本地 UI 也属于一个客户端实例。

Agent 本地 UI 能力：

- 接收当前机器产生的审批提醒。
- 查看本机当前会话和审批详情。
- 直接批准、拒绝或自定义回复。
- 显示远程手机或 Web 端已经处理的结果。
- 在本地用户直接操作终端后，上报本地接管结果。

Agent 本地处理规则：

- Agent 本地审批也必须经过服务端 API 创建审批动作。
- Agent 不能绕过服务端直接把远程审批标记为已处理。
- Agent 本地 UI 使用用户登录身份提交审批动作。
- Agent 回写 PTY 使用设备身份 ACK 投递结果。

这样可以区分：

- 谁做了审批决策。
- 哪个客户端提交了决策。
- 哪台机器执行了回写。

## 8. 权限需求

用户能查看和处理审批的条件：

- 用户属于审批所在租户。
- 用户角色具备审批权限。
- 用户对目标设备有访问权限，或设备策略允许租户级审批。

设备访问权限：

- 设备 owner 默认可查看和审批。
- 租户 admin 默认可查看和审批。
- approver 角色可按设备授权范围处理审批。
- viewer 只能查看，不能处理。

## 9. 审计需求

每次审批闭环至少记录：

- 审批由哪个 Agent 会话产生。
- 哪些客户端实例收到通知。
- 谁提交了审批动作。
- 通过哪个客户端类型提交。
- 提交内容是什么，脱敏后保存。
- 服务端是否成功下发到 Agent。
- Agent 是否成功写回 CLI。
- 其他客户端是否收到同步事件。

审计不要求记录每个客户端已读状态作为强一致数据，但需要能追踪通知发送和失败。
