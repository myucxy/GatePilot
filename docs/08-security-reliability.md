# 安全与可靠性设计

## 1. 安全边界

系统边界：

- Agent 运行在用户本机，能看到本地 CLI 输出，但只应上传审批相关内容。
- 服务端是审批事实来源，保存审批状态、审计和投递记录。
- Web/Mobile 只通过服务端访问数据，不直连 Agent。
- Agent 本地 UI 也必须通过服务端提交审批动作，不能绕过服务端状态机。
- Push 只做提醒，不承载敏感审批内容。

## 2. 身份与鉴权

### 2.1 用户鉴权

- 使用 OIDC 登录。
- 服务端校验 Access Token 的 issuer、audience、签名和过期时间。
- 用户在租户内的权限由 `tenant_members` 决定。
- 管理接口必须校验租户角色。
- Web、移动端、Agent 本地 UI 都必须注册客户端实例，审批动作需要带客户端实例 ID。

### 2.2 设备鉴权

- Agent 注册使用一次性激活码。
- 激活码只保存哈希，默认 10 分钟过期。
- 设备令牌只保存哈希。
- 设备令牌支持轮换和吊销。
- 禁用设备后立即拒绝新连接，并主动关闭现有连接。
- 设备身份只能证明“哪台机器在连接和回写”，不能证明“哪个用户做了审批”。

### 2.3 连接安全

- 所有生产连接必须使用 TLS。
- WebSocket 连接必须先认证再订阅或上报。
- Gateway 对认证失败、心跳异常、消息过大、频率异常进行限流和断开。
- 多客户端同步消息只发送给有租户权限和设备访问权限的客户端实例。

## 3. 敏感信息处理

### 3.1 终端输出脱敏

Agent 上传前执行第一层脱敏，服务端入库前执行第二层脱敏。

默认脱敏模式：

- API Key 常见格式。
- Bearer Token。
- GitHub/GitLab Token。
- OpenAI/Anthropic 等常见服务 Key。
- 私钥块。
- URL 中的用户名、密码、token query。
- 本地用户目录可按配置哈希化。

脱敏结果：

- 入库字段使用 `*_redacted`。
- 原始内容默认不入库。
- 如企业需要保留原文，必须启用单独配置，并使用字段级加密。

### 3.2 Push 内容限制

Push Payload 禁止包含：

- 原始 prompt。
- 终端上下文。
- 命令行原文。
- token、路径、用户名等敏感内容。

Push 只携带 `approval_id`、`risk_level`、`event_type` 和租户 ID。

### 3.3 前端展示

- 默认展示脱敏内容。
- 审计导出同样只导出脱敏内容。
- 管理员查看敏感上下文需要单独权限和审计记录。
- 已处理审批必须展示处理用户、客户端类型、处理时间和脱敏后的处理内容。

## 4. 审批可靠性

### 4.1 事实来源

PostgreSQL 是审批状态最终事实来源。Redis、WebSocket、Push 都不是最终事实来源。

### 4.2 幂等

必须幂等的动作：

- Agent 上报审批事件。
- 用户提交审批决策。
- Agent 本地 UI 提交审批决策。
- Worker 执行超时拒绝。
- 服务端下发审批决策。
- Agent 回传投递 ACK。

幂等键规则见 `docs/07-api-protocol.md`。

### 4.3 断线恢复

Agent 本地必须保存：

- 未确认上报的审批事件。
- 已收到但未成功 ACK 的投递。
- 会话状态变更记录。

重连后：

- Agent 补发未确认事件。
- 服务端返回已有审批单。
- Gateway 重放未 ACK 的投递。

### 4.4 超时策略

- 每个审批单必须有 `expires_at`。
- 超时默认拒绝。
- 超时拒绝也必须投递给 Agent。
- 投递失败不会回滚审批决策，但会进入 `delivery_failed` 并告警。

## 5. 风险控制

### 5.1 自动批准限制

- 自动批准默认关闭。
- 高风险和 critical 事件默认禁止全局自动批准。
- 自动批准策略必须有过期时间和审计备注。
- 自动批准命中必须写审计日志。

### 5.2 本地接管

本地用户可以直接在终端输入。系统必须处理远程审批和本地输入冲突：

- 如果远程审批尚未决策，Agent 上报 `approval.superseded`，服务端取消审批。
- 如果远程审批已经投递，记录冲突审计。
- 审计详情必须包含最终状态和 Agent 观察到的会话变化。

### 5.3 多端并发处理

同一个审批可能同时出现在多个手机、Web 页面和 Agent 本地 UI。

要求：

- 服务端使用乐观锁保证只有一个决策成功。
- 失败端必须收到明确冲突响应，并展示最终处理方。
- 同步消息必须包含处理客户端类型，但不能泄露其他端的敏感设备标识。
- 审计记录需要保存处理方用户、客户端实例、客户端类型和请求 ID。

### 5.4 消息大小限制

建议默认限制：

- 单条 WebSocket 消息最大 256 KB。
- 单个 prompt 文本最大 32 KB。
- 上下文片段最大 64 KB。
- 超出内容截断并标记 `truncated=true`。

## 6. 可观测性

### 6.1 Trace

每个审批链路从 Agent 检测到最终 ACK 使用同一个 `trace_id`。

链路节点：

- `approval.detected`
- `approval.created`
- `policy.evaluated`
- `approval.decided`
- `delivery.created`
- `delivery.sent`
- `delivery.acked`

### 6.2 告警

必须告警：

- `delivery_failed` 数量异常。
- Agent 大量离线。
- 审批超时率异常。
- WebSocket 认证失败激增。
- Worker 队列积压。
- 数据库连接池耗尽。

### 6.3 审计完整性

审计日志不参与用户主流程回滚。主事务成功但审计异步失败时，必须记录补偿任务。

## 7. 数据保留与合规

- 审批记录长期保留。
- 审计日志默认至少 180 天。
- 输出片段默认短期保留，建议 7 到 30 天。
- 租户可配置更严格保留策略。
- 删除用户时保留审计主体的不可逆匿名 ID，避免破坏审计链路。

## 8. 灾难恢复

恢复优先级：

1. PostgreSQL。
2. 服务端密钥。
3. OIDC 配置。
4. Agent 重新连接和补发。
5. Redis 在线状态重建。

Redis 丢失后：

- 在线状态可以由 Agent 心跳重建。
- 待投递任务从 PostgreSQL 查询恢复。
- 审批最终状态不受影响。
